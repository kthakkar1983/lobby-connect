# Auth email templates — REQUIRED for invites & password resets

**Why this exists:** Supabase email links (invite, password recovery) return the
session in the URL **fragment** by default. A server-side route can't read a
fragment, so the old `/auth/callback` (PKCE `exchangeCodeForSession`) never
established a session — invited users were bounced to `/sign-in`, never reached
`/onboarding`, never set a password, and every later sign-in failed with
`invalid_credentials`. Manually-created users worked only because they skip the
email-link flow entirely.

**The fix (code, shipped):** a `/auth/confirm` route that calls
`verifyOtp({ token_hash, type })` and writes the session cookies onto the
returned redirect response (`apps/portal/app/auth/confirm/route.ts`). The
sign-out route was fixed the same way (it was a silent no-op before).

**The fix (dashboard, manual — MUST be done):** point the email templates at
`/auth/confirm` so the link carries a `token_hash`. In **Supabase Dashboard →
Authentication → Email Templates**, edit the link's `href` (replace
`{{ .ConfirmationURL }}`):

| Template | New link `href` |
|---|---|
| **Invite user** | `{{ .SiteURL }}/auth/confirm?token_hash={{ .TokenHash }}&type=invite&next=/onboarding` |
| **Reset Password** | `{{ .SiteURL }}/auth/confirm?token_hash={{ .TokenHash }}&type=recovery&next=/auth/update-password` |

(`{{ .SiteURL }}` must be the clean prod alias `https://lobby-connect-portal.vercel.app`, already set in URL Configuration.)

**Notes**
- After changing templates, **re-send** any invite/reset — links generated by the
  old template still use the broken flow.
- Re-inviting an existing email is blocked in-app (`lib/users/invite.ts`), so to
  re-test an existing user, hard-delete + invite, or invite a fresh address.
- The built-in Supabase email service is rate-limited ("testing only"). Configure
  **custom SMTP** (Resend/SendGrid/etc.) before relying on email at the hotel —
  tracked in `docs/v2-backlog.md`.
- `/auth/callback` (the old OAuth `?code=` handler) was **removed** in Phase 4
  (M4) — it never persisted a session. All email links (invite + recovery) now go
  through `/auth/confirm`, which exchanges the `token_hash` via `verifyOtp`.
