# Live captions for agents: design

**Date:** 2026-06-25 (scope locked) · first drafted 2026-06-23
**Status:** Approved — ready for implementation plan
**Target release:** v1.1
**Surface:** Agent + Admin portals (`apps/portal/`) only
**Migrations / new routes:** one new portal API route (short-lived STT token mint). **No migrations, no RLS changes.**

## Context

During the pilot an agent reported difficulty understanding some US regional dialects on calls.
This is not a one-agent issue: comprehension under accent, background noise, and a phone-quality
audio channel will help every agent and improve guest outcomes generally. The ask is **live,
on-screen captions of what the guest is saying**, shown to the agent in real time while on a call.

Two call paths exist, and the agent's browser sits on the receiving end of both:

- **Video calls (Agora kiosk).** The guest is at a tablet; the agent answers in the portal. Agent
  side = `components/video-call/video-call.tsx`.
- **Phone calls (Twilio).** The guest is on a phone with no screen. Agent side = the softphone
  (`components/softphone/softphone.tsx` → the shared in-call overlay).

**Scope decision (2026-06-25): agent-side only.** Captions render only on the agent's screen, for
both call types. The earlier draft also captioned the agent's words onto the kiosk for the guest
(two-way); that is **cut from v1.1** and recorded as a clean future seam (see Open items). Cutting
it removes the per-property toggle, a migration, kiosk plumbing, and any kiosk-facing consent line.

**Sequencing decision (2026-06-25): video first, phone second.** Both ship in v1.1. Video leads
because night call volume skews kiosk (more video calls than phone calls after hours), and because
video is the lower-risk integration — Agora exposes the guest's audio track via a known accessor,
whereas the Twilio remote-audio accessor needs a short spike to confirm. Video-first ships value
while phone's one unknown is de-risked in parallel.

## Goals

1. Real-time captions of the **guest's** speech on the **agent's** in-call screen, for both video
   and phone calls. Sub-second latency, ephemeral (not stored).
2. **Provider-agnostic build.** Speechmatics is the chosen default; the build must keep the vendor
   behind a single adapter so switching (or A/B-ing) is a one-module change.
3. **Validate accuracy live, decide later.** No upfront formal bake-off (there is no stored pilot
   audio to test against — v1 never recorded). Ship behind the swap seam, judge Speechmatics on
   real calls during prod smoke, swap if it underperforms.

## Non-goals (explicit)

- **Storing transcripts.** Captions are live and ephemeral. Nothing is written to the database,
  Storage, or logs. This is also the safest privacy posture (see Consent).
- **Captioning the agent onto the kiosk (two-way).** Cut from v1.1; future seam.
- **Translation / non-English.** The pilot problem is English dialects, not other languages.
  Recorded as a future seam, not built here.
- **A server-side transcript pipeline / realtime fan-out.** The architecture is deliberately
  client-side so we do not build the WebSocket/SSE push layer that decision #4 (20s polling, no
  subscriptions) avoids.
- **Mobile agent/admin portals.** Desktop-only in v1, unchanged.

## Provider choice

**Speechmatics is the chosen default for v1.1**, implemented behind the adapter (below) so it is
the only place the vendor name appears. It is the one shortlisted candidate explicitly engineered
for accent/dialect robustness — the most aligned with the actual problem. We do not run an upfront
bake-off; we validate live on real calls and keep the swap open.

Swap candidates kept warm behind the same adapter (current streaming list prices — **source-backed,
fetched 2026-06-23; not re-verified since, and cost is not the deciding factor**):

| Provider | Live price/min | Note |
|---|---|---|
| **Speechmatics** (chosen) | ~$0.0067 | Engineered for accent/dialect robustness. |
| **AssemblyAI** (Universal-Streaming) | ~$0.0025 (Pro ~$0.0075) | Cheapest; flags a ~10% in-region rise from 2026-07-01, avoidable via the `global` region. |
| **Deepgram** (Nova-3) | ~$0.006–0.008 | Fast, mature browser streaming + temp tokens. |

Sources: [Speechmatics](https://www.speechmatics.com/pricing) ·
[AssemblyAI](https://www.assemblyai.com/pricing) · [Deepgram](https://deepgram.com/pricing).

Set aside (pricier, not better for this — estimates from the 06-23 fetch): Azure AI Speech
(~$0.017/min), OpenAI realtime (~$0.017/min), Google Cloud STT (~$0.024/min, awkward 15s-chunk
billing). Self-hosted Whisper is batch-oriented, not built for low-latency live streaming.

**Interim aid (free, available today):** an agent can switch on **Chrome Live Caption** right now
while v1.1 is built. It can't be the product — it captions all system audio with no way to isolate
the guest, can't live in our UI, and is inconsistent across machines — but it's a zero-cost stopgap.

## Architecture (provider-agnostic, client-side)

The core idea: **the device that needs the caption already holds the audio it needs to caption.**
No server round-trip for the audio, no new realtime push layer.

The **agent's browser**, on either call type, already has the **guest's** remote audio as a
`MediaStreamTrack`:

- **Video:** Agora gives it as `user.audioTrack.getMediaStreamTrack()` after
  `client.subscribe(user, "audio")` — the existing `user-published` handler in `video-call.tsx`.
- **Phone:** the `@twilio/voice-sdk` `Call` exposes the remote stream. **Confirm the exact accessor
  in the Step-0 spike** (flagged, not assumed).

The browser streams that one track to Speechmatics' realtime WebSocket and renders the returned
partial/final transcripts as a **local caption overlay**. The transcript never touches Supabase,
the 20s polling layer, or any server we run. This is why the feature needs no realtime
subscriptions and does not violate decision #4.

The **only** server-side addition is a tiny portal API route that mints a **short-lived, scoped
token** for Speechmatics, so the long-lived API key is never shipped to the browser:

```
GET /api/captions/token   (portal, new)
  requireApiActor({ allow: ["AGENT", "ADMIN"] })   // OWNER rejected; deactivated → 403
  → returns { token, expiresAt }                    // scoped, minutes-TTL; never the raw key
```

### The adapter (the swap seam)

A single module, `lib/captions/provider.ts`, exposes:

```ts
type CaptionStream = {
  start(track: MediaStreamTrack, onPartial: (t: string) => void, onFinal: (t: string) => void): Promise<void>
  stop(): void
}
createCaptionStream(token: string): CaptionStream
```

Speechmatics is implemented behind this interface. Switching providers later means a new
implementation of this one module — nothing else changes.

## Per-surface design (agent-side only)

### Video calls — ship first

Caption the **guest** into the existing video overlay (`video-call.tsx`), the same overlay that
already shows the playbook. Start the caption stream on the guest's `audioTrack` once subscribed;
stop on `user-left` / call end (reuse the existing teardown). Render `<CaptionBand>` as a legible
band over the deep-navy `--color-call` video stage.

### Phone calls — ship second

Caption the **guest** in the shared audio in-call overlay
(`components/softphone/audio-call-overlay.tsx`, the 2026-06-18 surface). The overlay already owns
the in-call screen for both agent and admin (shared `Softphone`), so captions appear identically in
both portals by construction. Start the caption stream when `phase === "in-call"`, stop on teardown.
Render `<CaptionBand>` in the call card (exact placement is a small UI decision in the plan).

### Caption band behavior

`<CaptionBand>` is presentational and shared by both overlays. It shows interim **partials**
(de-emphasized) and replaces them with stable **finals** (solid); it is hidden when there is no
active track or no speech. A stalled/dropped provider socket makes the band go quiet — it never
crashes the overlay and never sits in the audio path, so the call itself is unaffected.

## Consent / privacy

Recording was never enabled in v1, and these captions are **live and never stored** — a materially
lower-risk category than recording. Agent-only scope lowers it further: captions are an internal
comprehension aid on the agent's screen, never shown to the guest, functionally like the agent
listening harder.

Still, some US state two-party-consent laws are written broadly enough to potentially touch
real-time transcription, and the service may operate across states. **This warrants a quick legal
review before public launch. It is not a pilot blocker. I am not a lawyer — this is a flag, not
advice.** Design choices that keep us on the safe side regardless: ephemeral by construction (no
persistence anywhere) and agent-assist framing (internal aid, not a guest-facing artifact).

## Cost

Trivial at pilot scale. At Speechmatics' ~$0.0067/min, even an aggressive ~400 captioned
minutes/night lands around **$80/month** (estimate, on assumed volume). Cost does not constrain the
choice.

## Step-0 spike (resolve before the dependent UI work)

Two unknowns, confirmed by a tiny spike, not assumed:

1. **Twilio remote-audio accessor** — how to obtain the guest `MediaStreamTrack` from the
   `@twilio/voice-sdk` `Call`. Gates the *phone* wiring only; video can proceed regardless.
2. **Speechmatics browser streaming + temp-token specifics** — confirm the realtime WebSocket
   accepts a browser-direct connection authenticated by a short-lived scoped token (the pattern the
   token route mints), and the partial/final message shape the adapter parses. Gates the adapter.

## Testing

- **Adapter** unit-tested with a mocked WebSocket: emits partials then finals; `stop()` closes
  cleanly; a dropped socket surfaces without crashing (band just goes quiet).
- **Token route** tested like the other API routes: `requireApiActor` gate (AGENT/ADMIN only, OWNER
  rejected, deactivated → 403); returns a token + expiry; never the raw key.
- **Caption band** jsdom render test: shows partials, replaces with finals, hidden when no track.
- **Manual prod smoke** (voice/video are prod-only): **video first** — call as agent → guest
  captions appear within ~1s and stop on hang-up; then phone, same; same as admin; verify no
  transcript is persisted (DB/audit/Storage clean).

## Success criteria

- On a live **video** call, the agent sees captions of the guest's speech within ~1s, accurate
  enough to disambiguate the phrases that matter (room numbers, names, yes/no, addresses); captions
  stop and clear on call end.
- Same on a live **phone** call.
- Captions appear identically for AGENT and ADMIN; OWNER cannot obtain a token.
- Nothing is persisted — DB, audit, and Storage stay clean after captioned calls.
- The provider lives only in `lib/captions/provider.ts`; a swap touches no other file.

## Risks / mitigations

- **Speechmatics isn't good enough on our dialects.** → Swap seam + live validation; switch to
  AssemblyAI/Deepgram is a one-module change.
- **Twilio remote-audio accessor differs from assumed.** → Step-0 spike confirms it before the
  phone wiring; video (shipping first) is unaffected.
- **Caption latency/jitter degrades the call.** → Captions are a passive overlay on a track we
  already have; they never sit in the audio path, so a slow caption can't delay or break the call.
- **Exposing the STT key.** → Browser only ever receives a short-lived scoped token from our route;
  the key stays server-side.
- **Consent exposure.** → Ephemeral-by-construction + agent-only framing + legal review before
  public launch.

## Open items / future seams

- **Two-way kiosk captions** (caption the agent onto the kiosk for the guest). The cut v1.1 scope;
  re-enabling it adds a per-property `kiosk_captions_enabled` toggle (owner column-guard pattern,
  same shape as `kiosk_cta_style` migration 0015), kiosk plumbing, and an optional kiosk disclosure
  line. The adapter and `<CaptionBand>` are reused as-is on the kiosk side.
- **Translation** for non-English guests (Speechmatics offers it). The adapter interface can grow a
  target-language option.
- **Agora cloud STT** as an alternative video-path implementation if we ever want server-side
  generation/distribution or built-in translation (documented alternative, not chosen — browser-side
  single-provider is simpler and covers both call types).
- **Saving transcripts** (e.g. a call summary attached to the call record) would be a deliberate
  product + legal decision, explicitly out of scope here.
