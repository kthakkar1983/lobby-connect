## The v1.2.1 bug + fix (root cause, for context)

**Symptom:** kiosk video call rang the admin fast, but after the 120s no-answer apology the admin banner kept ringing **~60s** longer.

**Root cause (systematic-debugging):** prod `calls` row showed the call finalized to `NO_ANSWER` on time (`ring_to_end_secs=123`), so the clear *should* have pushed. But the 4 publisher routes fired the broadcast as a bare **`void broadcastCallsChanged(...)` after returning the response** — and **Vercel does not guarantee post-response detached work runs before the serverless function freezes**, so the call-ended clear-broadcast was dropped; only the 60s fallback poll cleared the banner. (call-started's ring-push happened to win its freeze race — that asymmetry, same channel/subscriber, isolated it to the un-awaited send.)

**Fix:** `after()` from `next/server` (waitUntil-backed) keeps the function alive until the broadcast completes; still non-blocking. Regression-guarded: each route test asserts the broadcast is scheduled via `after()`, not `void`. Full lesson in `MEMORY.md` → `build-quirks.md`.
