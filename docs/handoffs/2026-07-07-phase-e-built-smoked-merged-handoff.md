### THE BUG found + fixed live: RustDesk launch killed the video call
- **Symptom:** pressing Connect during a live video call ended the call — kiosk returned home, both feeds black, agent overlay stayed mounted.
- **Root cause (confirmed):** `launchRustdesk` used `window.location.assign("rustdesk://…")` on the **main window**, which fires the page's pagehide/unload → tears down the live **LiveKit WebRTC PeerConnections** → then cancels the navigation once the OS hands off to RustDesk (so React survives, overlay stays, but media is dead + the kiosk sees the agent drop → ends the call). Diagnostic that isolated it: **the call survives backgrounding** (the DocPiP tile stays up — the production "work in RustDesk, watch the guest on the tile" model is sound) **and dies ONLY on the launch**; server calls rows showed COMPLETED at ~9-10s (too fast for an occlusion/ICE timeout — an abrupt teardown).
- **Fix** (`13acedb`, this session, **post-whole-branch-review**): launch the deep link through a **transient hidden iframe** — a subframe navigation hands `rustdesk://` to the OS but structurally cannot unload the top document, so the video connections are untouched. (An anchor-without-target navigates the top frame too — same bug; the iframe is the safe path.) Regression test `tests/lib/remote-access/launch-rustdesk.test.ts` pins the iframe mechanism (a revert to a top-level nav fails it). **Kumar re-tested live: the video now survives Connect AND RustDesk still launches.** 652 node + 135 jsdom green.
