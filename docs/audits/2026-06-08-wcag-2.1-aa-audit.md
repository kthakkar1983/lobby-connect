# WCAG 2.1 Level A + AA Conformance Audit — Lobby Connect

**Created:** 2026-06-08 (session 12). **Scope:** `apps/portal` (Next.js — agent/admin/owner/auth)
+ `apps/kiosk` (Vite). **Standard:** WCAG 2.1, Level A + AA.
**Method:** code-level audit (rubric: `design:accessibility-review` skill) + computed contrast
ratios (sRGB relative-luminance formula). **Drives:** Stage 3 a11y remediation (Task 9 of
`docs/plans/2026-06-08-stage3-states-motion-a11y-copy.md`).

This is the Stage 3 §5.1 deliverable. Each applicable success criterion is recorded Pass / Fail /
N/A with evidence; every Fail carries a remediation note. The Fail list is the authoritative
remediation backlog.

## 0. Headline

The Stage 1/2 work left the **structural** layer in good shape — labels, keyboard operability,
focus-visible rings, target sizes, and ARIA roles largely pass. The real failures are
**contrast (1.4.3 / 1.4.11)** on several locked Stage 0 brand tokens, plus one Level A structural
gap (**2.4.1** skip link). Two contrast fixes touch the locked brand and need a product decision
(the coral CTA and the input-border hairline); the rest are mechanical.

## 1. Perceivable

### 1.1.1 Non-text content (A) — PASS (1 info)
Icon-only controls have accessible names: softphone, password toggle (`aria-label`), sidebar
toggle (`aria-label="Toggle Sidebar"`), table action menus (`aria-label="Actions"`), kiosk
`CallControls` (`aria-label={label}`), `RecordingNotice` close (`aria-label="Close"`),
availability `Switch` (`aria-label="Accept calls for …"`). Decorative icons/seam elements carry
`aria-hidden="true"` (27+ instances).
- **INFO:** `apps/kiosk/src/screens/Home.tsx:38` — property logo `alt=""`. Correct *if* decorative
  (it sits beside a text label). Confirm intent; if it's meaningful branding, give it a real alt.

### 1.3.1 Info & relationships (A) — PASS
All forms associate labels (explicit `htmlFor`/`id` or implicit wrap): sign-in, onboarding,
forgot/update-password, property form, user create/edit, kiosk-content edit, incident resolve.
Tables use semantic `<th>`/`<td>`. (Minor consistency: sign-in/forgot use implicit wrap vs.
explicit `htmlFor` elsewhere — normalize for consistency, not a failure.)

### 1.4.1 Use of color (A) — PASS
State is never color-only: status pills carry text, presence dots pair with labels, the line
beacon has `role="status"` + label, errors use icon + `role="alert"`. Carry this rule into the new
Stage 3 empty/error states.

## 2. Operable

### 2.1.1 Keyboard / 2.1.2 No trap (A) — PASS
Portal interactions are native `<button>`/`<a>` or Radix primitives (Dialog/Sheet/Select/
DropdownMenu) — fully keyboard-operable, no traps. No `onClick` on bare `div`/`span`. Kiosk is
touch-first (`type="button"` handlers) — not keyboard-dependent by design.

### 2.4.1 Bypass blocks (A) — **FAIL**
No skip-to-content link in `(admin)`, `(agent)`, `(owner)` layouts. **Remediation:** add an
`sr-only focus:not-sr-only` "Skip to content" anchor as the first focusable element in each authed
layout, targeting a `<main id="main">`.

### 2.4.2 Page titled (A) — PASS
Root `metadata.title = "Lobby Connect"`; routes carry visible headings. (Per-route `<title>` via
`metadata` exports would be a nice-to-have, not a failure.)

### 2.4.3 Focus order (A) — PASS · 2.4.7 Focus visible (AA) — PASS
Every `outline-none` is paired with a `focus-visible:ring-2 ring-ring ring-offset-2
ring-offset-background` replacement (button/input/select/textarea/switch). DOM order is logical.

### 2.4.6 Headings & labels (AA) — PARTIAL
Card sections use `<h2>`; most pages have an `<h1>`. **Fail (minor):** agent dashboard
(`app/(agent)/agent/page.tsx`) and owner home (`app/(owner)/owner/page.tsx`) have no `<h1>` — the
greeting is styled text, not a heading. **Remediation:** add a visible or `sr-only` `<h1>`.

### 2.5.8 Target size (minimum, AA, 24px) — PASS · 2.5.5 (AAA, 44px) — PASS on kiosk
Portal interactive targets ≥ 24px (smallest button `size-6`/h-8). Kiosk controls are 56px
(`size-14`), CTA huge, RecordingNotice buttons 56px+. Meets AA everywhere; kiosk meets AAA.

## 6. Open decisions — RESOLVED (session 12)

**D1 — the coral CTA → KEEP coral, ENLARGE the CTA text.** `--color-accent-strong` stays `#E05A39`
(brand unchanged). To clear 1.4.3, every **white-on-coral label** must qualify as WCAG *large text*:
**≥18.66px (1.1667rem) AND bold (≥700)**, or ≥24px at any weight. At large text the existing 3.69:1
passes (≥3.0). Applies to the primary `Button` coral variant, softphone **Hang up**, video **End**,
kiosk **Continue** (already `text-lg semibold` → bump to ≥18.66px + bold). **Where a coral button
cannot reasonably be that large** (e.g. a dense `size="sm"` table-row action), switch *that* button
to the **navy** fill (`primary`, white text 10.29:1 — passes at any size) rather than enlarge it.
Coral **text/links** (the separate MED finding, ≈13 sites) are *not* CTAs and can't be "large" —
they still move to the new AA-safe `--color-accent-text #C85033`.

**D2 — input borders → DARKEN to `#919598`.** `--color-input` becomes `#919598` (3.01:1 vs white —
full AA 1.4.11). Scope to **form controls only** (`input`/`textarea`/`select` use `border-input`);
`--color-border #E1E7EC` is unchanged so decorative dividers/card hairlines stay calm. Verify no
form control relies on `border-border` (would miss the fix) and no decorative element uses
`border-input` (would get unintentionally heavy).

(Both decisions affect only the named tokens/sites; everything else in §5 is mechanical.)
